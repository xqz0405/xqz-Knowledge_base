# 多线程与多进程

## What — 是什么

> Python 多线程（threading）和多进程（multiprocessing）是操作系统级并发方案，前者共享内存但受 GIL 限制仅能并行 I/O，后者独立进程真正利用多核 CPU。concurrent.futures 提供统一的高级接口封装两者。

**核心概念：**

- **threading**：Python 标准库的线程模块，创建轻量级线程共享进程内存，适合 I/O 密集型任务（网络请求、文件读写）
- **multiprocessing**：标准库的进程模块，创建独立进程各自拥有 GIL，适合 CPU 密集型任务（计算、数据处理）
- **GIL 限制**：CPython 的全局解释器锁使同一时刻只有一个线程执行 Python 字节码，多线程无法并行 CPU 计算
- **concurrent.futures**：高级并发接口，ThreadPoolExecutor 和 ProcessPoolExecutor 统一了线程和进程的提交/结果获取模式
- **进程间通信（IPC）**：多进程不共享内存，需通过 Queue、Pipe、Value、Array、Manager 等机制通信

**核心架构：**

- 线程模型：主线程 → 创建子线程 → 共享进程内存（需锁保护）→ 子线程完成 join 回收
- 进程模型：主进程 → fork/spawn 子进程 → 各自独立内存空间 → IPC 通信 → 子进程完成 join 回收
- 执行器模型：Executor.submit() → Future 对象 → result() 获取结果

**关键特性：**

- threading：Thread、Lock、RLock、Semaphore、Event、Condition、Barrier
- multiprocessing：Process、Pool、Queue、Pipe、Value、Array、Manager
- concurrent.futures：ThreadPoolExecutor、ProcessPoolExecutor、as_completed、wait

## Why — 为什么

**适用场景：**

- 多线程：网络请求并发、文件 I/O 并发、GUI 应用后台任务、数据库查询并发
- 多进程：大数据计算、图像处理、机器学习数据预处理、PDF 生成、压缩解压
- 执行器：简单并发任务的快速实现，不需要精细控制线程/进程生命周期

**多线程 vs 多进程 vs 协程：**

| 维度 | 多线程 | 多进程 | 协程 |
|------|--------|--------|------|
| 并行能力 | I/O 并行，CPU 受 GIL 限制 | 真正多核并行 | I/O 并行，单线程 |
| 内存 | 共享，开销小（~8KB/线程） | 独立，开销大（~30MB/进程） | 共享，极小（~2KB/协程） |
| 启动速度 | 快 | 慢（fork/spawn） | 极快 |
| 通信方式 | 共享变量（需加锁） | Queue/Pipe/Manager | 共享变量（无需锁） |
| 适用类型 | I/O 密集 | CPU 密集 | I/O 密集（高并发） |
| 调试难度 | 中（竞态条件） | 低（独立空间） | 低（单线程） |
| 稳定性 | 一个线程崩溃影响整个进程 | 进程隔离，互不影响 | 单线程，最稳定 |
| 规模上限 | 数百线程 | 数十进程 | 数万协程 |

**优缺点：**

- ✅ 多线程优点：轻量、共享内存、I/O 并行
- ❌ 多线程缺点：GIL 限制 CPU 并行、竞态条件、死锁风险
- ✅ 多进程优点：真正多核并行、进程隔离稳定
- ❌ 多进程缺点：内存开销大、启动慢、IPC 复杂

## How — 怎么用

### 1. threading 多线程

```python
import threading
import time

# 方式一：函数式
def worker(name, seconds):
    print(f'{name} 开始')
    time.sleep(seconds)
    print(f'{name} 完成，耗时 {seconds}s')
    return f'{name} result'

# 创建线程
t1 = threading.Thread(target=worker, args=('线程A', 2))
t2 = threading.Thread(target=worker, args=('线程B', 3), daemon=True)  # 守护线程

t1.start()
t2.start()
t1.join()   # 等待 t1 完成
# t2 是守护线程，主线程结束时自动终止
print('主线程结束')

# 方式二：继承 Thread
class WorkerThread(threading.Thread):
    def __init__(self, name, seconds):
        super().__init__(name=name)
        self.seconds = seconds
        self.result = None

    def run(self):
        print(f'{self.name} 开始')
        time.sleep(self.seconds)
        self.result = f'{self.name} done'
        print(f'{self.name} 完成')

t = WorkerThread('自定义线程', 2)
t.start()
t.join()
print(f'结果: {t.result}')
```

**线程同步原语：**

```python
import threading

# Lock — 互斥锁（最常用）
lock = threading.Lock()
counter = 0

def increment(n):
    global counter
    for _ in range(n):
        with lock:  # 等价于 lock.acquire() + lock.release()
            counter += 1

threads = [threading.Thread(target=increment, args=(100000,)) for _ in range(10)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # 1000000（无锁则远小于此）

# RLock — 可重入锁（同一线程可多次 acquire）
rlock = threading.RLock()

def outer():
    with rlock:
        inner()

def inner():
    with rlock:  # 同一线程可以再次获取，不会死锁
        print('可重入锁正常工作')

# Semaphore — 信号量（控制并发数）
semaphore = threading.Semaphore(3)  # 最多 3 个线程同时执行

def limited_worker(name):
    with semaphore:
        print(f'{name} 获得许可')
        time.sleep(1)
        print(f'{name} 释放许可')

# Event — 事件通知
event = threading.Event()

def waiter():
    print('等待事件...')
    event.wait()  # 阻塞直到 set()
    print('事件触发，继续执行')

def notifier():
    time.sleep(2)
    event.set()  # 触发事件

# Condition — 条件变量（生产者-消费者）
condition = threading.Condition()
items = []

def producer():
    for i in range(5):
        with condition:
            items.append(i)
            print(f'生产: {i}')
            condition.notify()  # 通知一个消费者
        time.sleep(0.5)

def consumer():
    with condition:
        while not items:
            condition.wait()  # 等待通知
        item = items.pop(0)
        print(f'消费: {item}')

# Barrier — 屏障（所有线程到齐后继续）
barrier = threading.Barrier(3)  # 3 个线程到齐后一起继续

def phase_worker(name):
    print(f'{name}: 阶段1')
    barrier.wait()  # 等待所有线程到达
    print(f'{name}: 阶段2')
    barrier.wait()
    print(f'{name}: 完成')
```

### 2. multiprocessing 多进程

```python
import multiprocessing
import time

# 基础进程
def square(n):
    return n * n

if __name__ == '__main__':
    p = multiprocessing.Process(target=square, args=(5,))
    p.start()
    p.join()

# 进程池
def square(n):
    return n * n

if __name__ == '__main__':
    with multiprocessing.Pool(processes=4) as pool:
        # map — 批量提交，保持顺序
        results = pool.map(square, range(10))
        print(results)  # [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

        # map_async — 异步版
        result = pool.map_async(square, range(10))
        print(result.get(timeout=5))

        # imap — 懒加载迭代器
        for result in pool.imap(square, range(10)):
            print(result)

        # apply — 单个任务
        result = pool.apply(square, (5,))

        # apply_async — 异步单个任务
        result = pool.apply_async(square, (5,))
        print(result.get())

    # starmap — 多参数
    def add(a, b):
        return a + b

    with multiprocessing.Pool(4) as pool:
        results = pool.starmap(add, [(1, 2), (3, 4), (5, 6)])
        print(results)  # [3, 7, 11]
```

**进程间通信：**

```python
import multiprocessing

# Queue — 进程安全队列
def producer(q):
    for i in range(5):
        q.put(f'item-{i}')
    q.put(None)  # 哨兵值

def consumer(q):
    while True:
        item = q.get()
        if item is None:
            break
        print(f'消费: {item}')

if __name__ == '__main__':
    q = multiprocessing.Queue()
    p1 = multiprocessing.Process(target=producer, args=(q,))
    p2 = multiprocessing.Process(target=consumer, args=(q,))
    p1.start()
    p2.start()
    p1.join()
    p2.join()

# Pipe — 双向管道
def sender(conn):
    conn.send('hello from sender')
    msg = conn.recv()
    print(f'sender 收到: {msg}')
    conn.close()

def receiver(conn):
    msg = conn.recv()
    print(f'receiver 收到: {msg}')
    conn.send('hello from receiver')
    conn.close()

if __name__ == '__main__':
    parent_conn, child_conn = multiprocessing.Pipe()
    p1 = multiprocessing.Process(target=sender, args=(parent_conn,))
    p2 = multiprocessing.Process(target=receiver, args=(child_conn,))
    p1.start()
    p2.start()
    p1.join()
    p2.join()

# Value / Array — 共享内存
def increment(counter):
    with counter.get_lock():
        counter.value += 1

if __name__ == '__main__':
    counter = multiprocessing.Value('i', 0)  # 'i' = int
    processes = [multiprocessing.Process(target=increment, args=(counter,))
                 for _ in range(100)]
    for p in processes: p.start()
    for p in processes: p.join()
    print(counter.value)  # 100

# Manager — 托管对象（支持 dict/list 等）
def worker(d, l):
    d['key'] = 'value'
    l.append(42)

if __name__ == '__main__':
    with multiprocessing.Manager() as manager:
        shared_dict = manager.dict()
        shared_list = manager.list()
        p = multiprocessing.Process(target=worker, args=(shared_dict, shared_list))
        p.start()
        p.join()
        print(shared_dict)  # {'key': 'value'}
        print(shared_list)  # [42]
```

### 3. concurrent.futures 高级接口

```python
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor, as_completed
import time

# ThreadPoolExecutor — I/O 密集型
def fetch_url(url):
    """模拟网络请求"""
    time.sleep(1)
    return f'{url} → 200'

urls = [f'https://api.example.com/{i}' for i in range(10)]

# 方式一：map（保持顺序）
with ThreadPoolExecutor(max_workers=5) as executor:
    results = list(executor.map(fetch_url, urls))
    print(results)

# 方式二：submit + as_completed（谁先完成谁先处理）
with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(fetch_url, url): url for url in urls}
    for future in as_completed(futures):
        url = futures[future]
        try:
            result = future.result()
            print(f'{url}: {result}')
        except Exception as e:
            print(f'{url} 失败: {e}')

# ProcessPoolExecutor — CPU 密集型
def heavy_compute(n):
    """模拟 CPU 密集计算"""
    total = 0
    for i in range(n):
        total += i * i
    return total

if __name__ == '__main__':
    numbers = [10**7, 10**7, 10**7, 10**7]

    with ProcessPoolExecutor(max_workers=4) as executor:
        results = list(executor.map(heavy_compute, numbers))
        print(results)

# 带超时和回调
with ThreadPoolExecutor(max_workers=3) as executor:
    future = executor.submit(fetch_url, 'https://api.example.com/1')

    # 非阻塞检查
    print(future.done())    # False
    print(future.running()) # True

    # 带超时获取结果
    result = future.result(timeout=5)

    # 回调函数
    def on_complete(future):
        try:
            print(f'回调: {future.result()}')
        except Exception as e:
            print(f'回调异常: {e}')

    future2 = executor.submit(fetch_url, 'https://api.example.com/2')
    future2.add_done_callback(on_complete)
```

### 4. 实战：多线程爬虫 + 多进程计算

```python
"""多线程并发爬取 + 多进程并行处理"""
import time
import json
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# 模拟网络请求
def fetch_page(url):
    time.sleep(0.5)  # 模拟网络延迟
    return {'url': url, 'data': f'content of {url}', 'length': len(url) * 100}

# CPU 密集处理
def process_data(page):
    """模拟数据清洗和分析"""
    result = 0
    for i in range(10**6):
        result += i
    page['processed'] = True
    page['score'] = result % 100
    return page

def main():
    urls = [f'https://example.com/page/{i}' for i in range(20)]

    # 阶段1：多线程爬取（I/O 密集）
    print('阶段1: 多线程爬取...')
    start = time.perf_counter()
    with ThreadPoolExecutor(max_workers=10) as executor:
        pages = list(executor.map(fetch_page, urls))
    print(f'  爬取完成: {time.perf_counter() - start:.2f}s, {len(pages)} 页')

    # 阶段2：多进程处理（CPU 密集）
    print('阶段2: 多进程处理...')
    start = time.perf_counter()
    with ProcessPoolExecutor() as executor:
        results = list(executor.map(process_data, pages))
    print(f'  处理完成: {time.perf_counter() - start:.2f}s')

    # 输出
    for r in results[:3]:
        print(f"  {r['url']}: score={r['score']}")

if __name__ == '__main__':
    main()
```

### 5. 常见陷阱与解决方案

```python
# 陷阱1：竞态条件
import threading

counter = 0
lock = threading.Lock()

# ❌ 不安全
def unsafe_increment():
    global counter
    for _ in range(100000):
        counter += 1  # 非原子操作：read → modify → write

# ✅ 安全
def safe_increment():
    global counter
    for _ in range(100000):
        with lock:
            counter += 1

# 陷阱2：死锁
lock_a = threading.Lock()
lock_b = threading.Lock()

# ❌ 死锁：两个线程交叉获取锁
def task1():
    with lock_a:
        time.sleep(0.01)
        with lock_b:  # 等待 lock_b，但 task2 已持有
            pass

def task2():
    with lock_b:
        time.sleep(0.01)
        with lock_a:  # 等待 lock_a，但 task1 已持有
            pass

# ✅ 解决：统一锁获取顺序
def task1_safe():
    with lock_a, lock_b:  # 总是先 A 后 B
        pass

def task2_safe():
    with lock_a, lock_b:  # 同样先 A 后 B
        pass

# 陷阱3：Windows 下 multiprocessing 必须有 __main__ 守卫
# ❌ Windows 下会递归创建进程
# def main():
#     p = multiprocessing.Process(target=worker)
#     p.start()
# main()

# ✅ 必须用 if __name__ == '__main__'
if __name__ == '__main__':
    p = multiprocessing.Process(target=worker)
    p.start()

# 陷阱4：多进程不能传 lambda（不可 pickle）
# ❌
# pool.map(lambda x: x**2, range(10))

# ✅ 用顶层函数
def square(x):
    return x ** 2
pool.map(square, range(10))

# 陷阱5：守护线程中创建子线程
# 守护线程结束时，其子线程也被强制终止（即使非守护）
# 解决：避免在守护线程中做需要清理的操作
```

## 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 多线程 CPU 计算没有加速 | GIL 限制，同一时刻只有一个线程执行 Python 代码 | CPU 密集用 multiprocessing |
| counter 结果不对 | `+=` 非原子操作，竞态条件 | 加 `Lock` 或用 `atomic` 操作 |
| 死锁 | 两个线程交叉获取锁 | 统一锁获取顺序，使用 `with` 避免忘记释放 |
| Windows 下进程报错 | 缺少 `if __name__ == '__main__'` 守卫 | 加 `__main__` 守卫 |
| ProcessPoolExecutor 卡住 | 函数或参数不可 pickle | 使用顶层函数，避免 lambda/局部函数 |
| 进程内存占用过高 | 每个进程独立内存空间 | 用 Pool 控制进程数，共享大数据用 Manager |
| 线程数过多变慢 | 线程切换开销 + GIL 争抢 | 控制线程数（CPU核数×2~5），I/O密集用协程 |
| 子进程异常丢失 | 子进程异常不会自动传播 | `future.result()` 会 re-raise，`pool.map` 会传播 |

## 面试题

**Q1: Python 的 GIL 是什么？为什么多线程不能加速 CPU 密集型任务？**

> GIL（Global Interpreter Lock）是 CPython 中的全局互斥锁，任何线程执行 Python 字节码前必须获取 GIL，同一时刻只有一个线程执行 Python 代码。GIL 存在的原因：CPython 的内存管理（引用计数）不是线程安全的，GIL 简化了实现。影响：多线程无法并行执行 CPU 密集的 Python 代码——即使有 4 核 CPU、4 个线程，同一时刻也只有 1 个线程在执行。但 I/O 操作（网络、文件、sleep）会释放 GIL，所以多线程可以加速 I/O 密集型任务。解决方案：(1) CPU 密集用 multiprocessing（每个进程独立 GIL）；(2) 用 C 扩展/Numpy（在 C 层释放 GIL）；(3) 用 asyncio 协程（单线程 I/O 并发）。

**Q2: threading.Lock 和 threading.RLock 有什么区别？**

> Lock 是普通互斥锁，同一线程第二次 acquire 会死锁。RLock 是可重入锁，同一线程可以多次 acquire，每次 acquire 需要对应一次 release，全部 release 后其他线程才能获取。RLock 内部维护 owner 线程和计数器。适用场景：Lock 适用于简单互斥（绝大多数情况）；RLock 适用于递归函数或类方法中需要多次获取同一把锁的场景（如方法 A 调用方法 B，两者都需要同一把锁）。RLock 性能略低于 Lock（额外的计数和判断），非必要不用。

**Q3: concurrent.futures 的 submit 和 map 有什么区别？**

> (1) **submit**——提交单个任务，返回 Future 对象。可对每个 Future 独立处理（回调、超时、异常捕获）。适合任务参数不同、需要精细控制的场景；(2) **map**——批量提交，返回结果迭代器，保持提交顺序。写法简洁，一行代码搞定批量提交和结果收集。适合任务参数来自同一可迭代对象、结果顺序重要的场景。关键区别：map 遇到异常会立即抛出，submit 的异常存储在 Future 中，调用 `result()` 时才抛出。需要按完成顺序处理结果（非提交顺序）时，用 submit + `as_completed()`。

**Q4: 多进程的进程间通信有哪些方式？各自适用什么场景？**

> 四种方式：(1) **Queue**——进程安全队列，最常用，适合生产者-消费者模式，支持 put/get，先进先出；(2) **Pipe**——双向管道，两个端点，适合一对一通信，性能比 Queue 高但不支持多消费者；(3) **Value/Array**——共享内存（ctypes 实现），适合共享简单数据（整数、浮点、数组），需用 `get_lock()` 保证安全，性能最高但只支持基本类型；(4) **Manager**——托管对象，支持 dict/list/Lock/Queue 等任意 Python 对象，最灵活但性能最低（每次访问都通过 IPC 代理）。选择原则：简单队列用 Queue，一对一高性能用 Pipe，共享数值用 Value/Array，复杂共享状态用 Manager。

**Q5: 如何选择多线程、多进程、协程？**

> 决策树：(1) 任务是 CPU 密集还是 I/O 密集？CPU 密集→多进程，I/O 密集→多线程或协程；(2) 并发规模多大？少量（<100）→多线程，大量（>1000）→协程；(3) 是否需要真正并行？是→多进程，否→协程；(4) 是否需要共享内存？是→多线程（加锁），否→多进程或协程；(5) 是否需要进程隔离？是→多进程。典型组合：网络爬虫（I/O + 高并发）→协程；数据处理/ML（CPU）→多进程；Web 服务后台任务（I/O + 少量）→多线程；混合场景→多进程 + 每个进程内协程。

**Q6: ProcessPoolExecutor 和直接用 multiprocessing.Pool 有什么区别？**

> (1) **API 风格**——ProcessPoolExecutor 是 concurrent.futures 的一部分，和 ThreadPoolExecutor 统一接口（submit/map/result），multiprocessing.Pool 有自己的 API（apply/apply_async/map/map_async）；(2) **返回值**——Executor 返回 Future 对象，Pool 返回 AsyncResult 对象，两者接口不同；(3) **上下文管理**——Executor 支持 `with` 语句自动清理，Pool 的 `with` 在 Python 3.3+ 才支持；(4) **取消能力**——Future 可以 cancel（未开始的任务），AsyncResult 不行；(5) **异常处理**——Executor 的异常在 `result()` 中抛出，Pool 的异常在 `get()` 中抛出。推荐：新代码用 ProcessPoolExecutor（统一 API + 更现代），需要 Pool 独有功能（如 imap_unordered）时用 Pool。

**Q7: 什么是守护线程/进程？有什么注意事项？**

> 守护线程（`daemon=True`）：主线程结束时，所有守护线程被强制终止，不等待完成。适用于后台监控、日志刷新等"可丢弃"任务。注意：(1) 守护线程中不要做文件写入、数据库提交等需要清理的操作（可能被中途杀死）；(2) 守护线程中创建的子线程默认也是守护线程；(3) 必须在 `start()` 之前设置 `daemon=True`。守护进程类似，主进程退出时守护进程被终止。多进程中的守护进程不能创建子进程（否则守护进程终止时子进程变成孤儿进程）。

**Q8: 如何调试多线程/多进程的并发问题？**

> (1) **竞态条件**——最难调试，表现为结果不确定。方法：加日志记录线程 ID 和操作顺序；使用 `threading.get_ident()` 追踪；临时加 `time.sleep()` 放大竞态窗口便于复现；(2) **死锁**——程序卡住不动。方法：`threading.timeout` 设超时检测；用 `with lock` 替代手动 acquire/release；检查锁获取顺序是否一致；(3) **性能分析**——`threading.active_count()` 查看活跃线程数；`concurrent.futures` 的 Future 对象可统计耗时；(4) **多进程调试**——每个进程独立，可在子进程中加 print/logging；`multiprocessing.log_to_stderr()` 开启进程模块日志；(5) **工具**——`pdb` 调试单线程逻辑；`faulthandler` 捕获段错误；`objgraph` 排查内存泄漏。
