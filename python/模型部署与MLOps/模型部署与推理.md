# 模型部署与推理

## What — 模型部署与推理是什么

模型部署是将训练好的模型投入生产环境提供服务的过程，推理是模型对新数据做出预测的执行阶段。训练关注模型精度，部署关注延迟、吞吐、资源占用和可靠性——这两者的优化方向往往不同。

核心挑战：

| 挑战 | 说明 |
|------|------|
| 延迟 | 单次推理的响应时间 |
| 吞吐 | 单位时间处理的请求数 |
| 资源 | GPU/CPU/内存成本 |
| 模型大小 | 存储和传输开销 |
| 版本管理 | 模型更新不中断服务 |
| 监控 | 检测性能衰退和数据漂移 |

```python
# 训练与部署的差异
# 训练：精度优先，可以用大模型、长训练、大量GPU
# 推理：效率优先，需要低延迟、高吞吐、低成本
```

## Why — 为什么部署优化很重要

一个训练精度 99% 但推理耗时 5 秒的模型在生产中无法使用。部署优化可以让模型在不损失精度的情况下快 2-20 倍、小 4-8 倍，直接影响用户体验和成本。

## How — 部署与优化方法

### 1. 模型导出格式

```python
import torch
import torch.nn as nn
import numpy as np

model = nn.Sequential(nn.Linear(20, 10), nn.ReLU(), nn.Linear(10, 2))
model.eval()
dummy = torch.randn(1, 20)

# === PyTorch 原生保存 ===
torch.save(model.state_dict(), 'model.pt')

# === TorchScript ===
# 方式1：tracing（记录一次前向的运算图）
scripted = torch.jit.trace(model, dummy)
scripted.save('model_traced.pt')

# 方式2：scripting（分析 Python 源码）
# scripted = torch.jit.script(model)  # 需要代码兼容
loaded = torch.jit.load('model_traced.pt')
print(loaded(dummy))

# === ONNX（跨框架标准格式） ===
torch.onnx.export(
    model, dummy, 'model.onnx',
    input_names=['input'], output_names=['output'],
    dynamic_axes={'input': {0: 'batch'}, 'output': {0: 'batch'}},
    opset_version=17
)

# 验证 ONNX 模型
import onnx
onnx_model = onnx.load('model.onnx')
onnx.checker.check_model(onnx_model)
print("ONNX 模型验证通过")

# === 验证导出一致性 ===
with torch.no_grad():
    pt_out = model(dummy)

import onnxruntime as ort
sess = ort.InferenceSession('model.onnx')
ort_out = sess.run(None, {'input': dummy.numpy()})

np.testing.assert_allclose(pt_out.numpy(), ort_out[0], rtol=1e-3, atol=1e-5)
print("PyTorch 和 ONNX Runtime 输出一致")
```

| 格式 | 跨框架 | 优化 | 部署生态 | 推理速度 |
|------|--------|------|---------|---------|
| PyTorch .pt | 否 | 有限 | TorchServe | 基线 |
| TorchScript | 否 | 中等 | 嵌入式 | 1.2-2x |
| ONNX | 是 | 丰富 | ORT/TRT/OpenVINO | 2-5x |
| TensorRT | 否(NVIDIA) | 最强 | NVIDIA GPU | 3-10x |
| OpenVINO | 否(Intel) | 强 | Intel CPU/GPU | 2-5x |

### 3. 模型量化

量化将模型权重从 FP32 降为 FP16/INT8/INT4，大幅减少模型大小和推理延迟。

```python
import torch
import torch.nn as nn
import numpy as np

model = nn.Sequential(
    nn.Linear(20, 64), nn.ReLU(),
    nn.Linear(64, 32), nn.ReLU(),
    nn.Linear(32, 2)
)
model.eval()

# === PyTorch 动态量化（最简单） ===
quantized_model = torch.quantization.quantize_dynamic(
    model, {nn.Linear}, dtype=torch.qint8
)

# 比较大小
def model_size(m):
    return sum(p.numel() * p.element_size() for p in m.parameters()) / 1024

print(f"原始: {model_size(model):.1f}KB")
print(f"量化: {model_size(quantized_model):.1f}KB")

# === PyTorch 静态量化（更精确） ===
model_q = nn.Sequential(
    nn.Linear(20, 64), nn.ReLU(),
    nn.Linear(64, 32), nn.ReLU(),
    nn.Linear(32, 2)
)
model_q.qconfig = torch.quantization.get_default_qconfig('x86')
prepared = torch.quantization.prepare(model_q)

# 用校准数据跑几个 batch
with torch.no_grad():
    for _ in range(100):
        prepared(torch.randn(32, 20))

quantized_static = torch.quantization.convert(prepared)

# === ONNX 量化 ===
from onnxruntime.quantization import quantize_dynamic, QuantType

quantize_dynamic(
    'model.onnx', 'model_quant.onnx',
    weight_type=QuantType.QInt8
)

# === FP16 量化（GPU 推理常用） ===
from onnxconverter_common import float16

import onnx
model_onnx = onnx.load('model.onnx')
model_fp16 = float16.convert_float_to_float16(model_onnx)
onnx.save(model_fp16, 'model_fp16.onnx')

# === 量化精度对比 ===
import onnxruntime as ort

sess_fp32 = ort.InferenceSession('model.onnx')
sess_int8 = ort.InferenceSession('model_quant.onnx')

test_data = np.random.randn(100, 20).astype(np.float32)
out_fp32 = sess_fp32.run(None, {'input': test_data})[0]
out_int8 = sess_int8.run(None, {'input': test_data})[0]

# 精度损失
max_diff = np.abs(out_fp32 - out_int8).max()
cos_sim = np.sum(out_fp32 * out_int8) / (np.linalg.norm(out_fp32) * np.linalg.norm(out_int8))
print(f"最大差异: {max_diff:.6f}, 余弦相似度: {cos_sim:.6f}")

# 速度对比
import time
for name, sess in [('FP32', sess_fp32), ('INT8', sess_int8)]:
    start = time.perf_counter()
    for _ in range(1000):
        sess.run(None, {'input': test_data[:1]})
    elapsed = time.perf_counter() - start
    print(f"{name}: {elapsed:.3f}s (1000次)")
```

| 量化类型 | 精度损失 | 加速比 | 模型缩小 | 适用 |
|----------|---------|--------|---------|------|
| FP32→FP16 | 极小 | 1.5-2x | 2x | GPU |
| FP32→INT8 动态 | 小 | 2-3x | 4x | CPU |
| FP32→INT8 静态 | 中 | 2-4x | 4x | CPU/边缘 |
| INT8→INT4 | 较大 | 3-5x | 8x | 极端资源受限 |

### 4. TensorRT 加速

```python
# === TensorRT 优化（NVIDIA GPU） ===
# 1. ONNX → TensorRT engine

# 使用 trtexec 命令行工具（最简单）
# trtexec --onnx=model.onnx --saveEngine=model.engine --fp16

# 使用 Python API
import tensorrt as trt

logger = trt.Logger(trt.Logger.WARNING)
builder = trt.Builder(logger)
network = builder.create_network(1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH))
parser = trt.OnnxParser(network, logger)

with open('model.onnx', 'rb') as f:
    parser.parse(f.read())

config = builder.create_builder_config()
config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 << 30)  # 1GB
config.set_flag(trt.BuilderFlag.FP16)  # 启用 FP16

engine = builder.build_serialized_network(network, config)
with open('model.engine', 'wb') as f:
    f.write(engine)

# === TensorRT 推理 ===
import pycuda.driver as cuda
import pycuda.autoinit

def trt_inference(engine_path, input_data):
    with open(engine_path, 'rb') as f:
        engine = trt.Runtime(logger).deserialize_cuda_engine(f.read())

    context = engine.create_execution_context()

    # 分配 GPU 内存
    input_shape = engine.get_binding_shape(0)
    output_shape = engine.get_binding_shape(1)

    input_mem = cuda.mem_alloc(input_data.nbytes)
    output_mem = cuda.mem_alloc(output_data.nbytes)

    bindings = [int(input_mem), int(output_mem)]

    # 拷贝输入 → GPU
    cuda.memcpy_htod(input_mem, input_data)

    # 执行推理
    context.execute_v2(bindings)

    # 拷贝输出 → CPU
    output = np.empty(output_shape, dtype=np.float32)
    cuda.memcpy_dtoh(output, output_mem)

    return output
```

# 1. 打包模型
# torch-model-archiver --model-name mymodel --version 1.0 \

# === FastAPI 自定义服务 ===
from fastapi import FastAPI
from pydantic import BaseModel
import numpy as np
import onnxruntime as ort
import time

app = FastAPI()
sess = ort.InferenceSession('model.onnx')

class PredictRequest(BaseModel):
    features: list[list[float]]

@app.post('/predict')
async def predict(req: PredictRequest):
    input_data = np.array(req.features, dtype=np.float32)
    results = sess.run(None, {'input': input_data})
    return {
        'predictions': results[0].tolist(),
        'latency_ms': 0  # 可加计时
    }

@app.get('/health')
async def health():
    return {'status': 'healthy'}

# 启动: uvicorn serve:app --host 0.0.0.0 --port 8000 --workers 4

# === 批量推理端点 ===
@app.post('/predict_batch')
async def predict_batch(req: PredictRequest):
    start = time.perf_counter()
    input_data = np.array(req.features, dtype=np.float32)
    results = sess.run(None, {'input': input_data})
    elapsed = (time.perf_counter() - start) * 1000
    return {
        'predictions': results[0].tolist(),
        'batch_size': len(req.features),
        'latency_ms': elapsed
    }

### 8. 监控与 A/B 测试

```python
# === 模型监控 ===
import numpy as np
from scipy import stats

class ModelMonitor:
    def __init__(self, reference_data, threshold=0.05):
        self.reference_mean = np.mean(reference_data, axis=0)
        self.reference_std = np.std(reference_data, axis=0)
        self.threshold = threshold
        self.prediction_log = []

    def check_data_drift(self, new_data):
        """检测输入数据漂移"""
        drift_detected = False
        for i in range(new_data.shape[1]):
            stat, p = stats.ks_2samp(
                (self.reference_data[:, i] - self.reference_mean[i]) / self.reference_std[i],
                (new_data[:, i] - self.reference_mean[i]) / self.reference_std[i]
            )
            if p < self.threshold:
                print(f"  特征 {i} 漂移检测: p={p:.4f}")
                drift_detected = True
        return drift_detected

    def check_prediction_drift(self, predictions):
        """检测预测分布漂移"""
        self.prediction_log.extend(predictions.tolist())
        if len(self.prediction_log) >= 1000:
            recent = np.array(self.prediction_log[-500:])
            old = np.array(self.prediction_log[:500])
            stat, p = stats.ks_2samp(old, recent)
            if p < self.threshold:
                print(f"  预测分布漂移: p={p:.4f}")
                return True
        return False

# === A/B 测试 ===
class ABTestRouter:
    def __init__(self, model_a, model_b, traffic_ratio=0.5):
        self.model_a = model_a
        self.model_b = model_b
        self.traffic_ratio = traffic_ratio
        self.results_a = []
        self.results_b = []

    def route(self, input_data, user_id):
        """基于用户ID哈希分配流量"""
        if hash(user_id) % 100 < self.traffic_ratio * 100:
            result = self.model_a.predict(input_data)
            self.results_a.append(result)
            return result, 'A'
        else:
            result = self.model_b.predict(input_data)
            self.results_b.append(result)
            return result, 'B'

    def compare(self):
        """比较两个模型的效果"""
        from scipy import stats
        if len(self.results_a) > 30 and len(self.results_b) > 30:
            stat, p = stats.ttest_ind(self.results_a, self.results_b)
            print(f"A/B测试: A均值={np.mean(self.results_a):.4f}, "
                  f"B均值={np.mean(self.results_b):.4f}, p={p:.4f}")
            return p < 0.05  # 差异显著
        return False
```

## 面试题

### 1. ONNX 和 TorchScript 的区别？

ONNX 是跨框架的开放标准，支持 PyTorch/TensorFlow/MXNet 等导出，用 ONNX Runtime/TensorRT/OpenVINO 等多种运行时推理，生态最广。TorchScript 是 PyTorch 原生方案，通过 trace 或 script 将模型转为静态图，只能在 PyTorch 环境运行。ONNX 优势是跨平台部署选择多，TorchScript 优势是支持更复杂的 Python 逻辑（script 模式）。实践建议：生产部署导出 ONNX → ONNX Runtime/TensorRT 推理，移动端用 CoreML/TF Lite。

### 2. 动态量化和静态量化怎么选？

动态量化在推理时实时将权重从 INT8 反量化为 FP32 再计算，激活值保持 FP32。静态量化在推理前就将权重转为 INT8，激活值也用 INT8（需要校准数据确定激活值的缩放因子）。动态量化更简单（不需要校准数据）、精度损失更小，但加速不如静态量化明显（仍需反量化操作）。静态量化更快但需要代表性校准数据，且可能精度损失较大。建议：NLP 模型（Embedding+Linear）用动态量化即可，CV 模型（Conv密集）用静态量化加速更明显。GPU 推理优先用 FP16 而非 INT8。

### 3. TensorRT 为什么比 ONNX Runtime 快？

TensorRT 在 ONNX 模型基础上做了三层优化：（1）**层融合**（Layer Fusion）——将多个算子合并为一个 GPU kernel，减少显存访问和 kernel 启动开销，如 Conv+BN+ReLU 融合为一个 kernel；（2）**精度校准**——INT8 量化时用校准数据确定最优缩放因子，最小化精度损失；（3）**内核自动调优**——针对具体 GPU 架构选择最优的 kernel 实现（如矩阵乘法的分块策略）。代价是 engine 与 GPU 型号绑定，换 GPU 需重新编译，且编译耗时较长。

### 4. 模型剪枝和量化的区别？

剪枝（Pruning）将不重要的权重置零，产生稀疏矩阵，需要稀疏计算硬件/软件支持才能加速。量化（Quantization）降低权重的数值精度（FP32→INT8），模型大小直接缩小4倍，所有硬件都能加速。剪枝可以和量化结合：先剪枝再量化（稀疏+低精度）。非结构化剪枝（逐元素置零）精度损失小但需要特殊稀疏引擎，结构化剪枝（整个通道/层置零）精度损失稍大但普通硬件就能加速。实践中量化比剪枝更常用，因为效果确定且无需特殊运行时支持。

### 5. 如何设计高可用的模型服务？

关键设计：（1）**负载均衡**——多实例部署 + Nginx/K8s Service 分流；（2）**健康检查**——/health 端点 + 心跳检测，异常实例自动摘除；（3）**优雅降级**——主模型超时时回退到轻量模型，保证响应时间；（4）**版本管理**——蓝绿部署或金丝雀发布，新版本逐步切流；（5）**自动扩缩**——基于 QPS/CPU/GPU 利用率自动增减实例（K8s HPA）；（6）**批处理**——将短时间内的多个请求合并为 batch 推理，提高 GPU 利用率。

### 6. 边缘设备部署的挑战和解决方案？

挑战：（1）算力有限（CPU为主，内存小）→ 量化到 INT8/INT4，用 OpenVINO/TF Lite；（2）模型太大（数百MB）→ 剪枝+量化+蒸馏，将模型压缩到 10MB 以内；（3）无网络 → 全部本地推理，模型嵌入 App；（4）电池功耗 → 选低精度计算，减少内存访问。解决方案：Intel 设备用 OpenVINO，ARM 设备用 TF Lite/NCNN，Apple 设备用 CoreML，NVIDIA 边缘用 TensorRT。知识蒸馏也是有效方法——用大模型做教师训练小模型。

### 7. 如何检测模型在生产中的性能衰退？

三种信号：（1）**数据漂移**——输入特征分布发生变化（如用户画像变化），用 KS 检验或 PSI 指标监控；（2）**预测漂移**——模型输出的分布发生变化，可能意味着输入分布或决策边界变了；（3）**标签漂移**——真实标签分布变化（如新类别出现），需要延迟反馈机制收集标签。实践方案：定期用新数据计算特征分布与训练数据的 PSI，超过阈值触发告警；记录预测分布并监控统计量变化；建立人工反馈通道收集真实标签，定期重评估模型指标。发现衰退后需重新训练或微调。

### 8. 批量推理和单条推理的权衡？

批量推理将多个请求合并为一个 batch 送入模型，GPU 利用率高、吞吐大，但单条延迟高（等batch凑齐）。单条推理立即响应、延迟低，但 GPU 利用率低、吞吐受限。权衡策略：（1）**动态批处理**——设最大等待时间（如10ms），到时间或凑齐 max_batch_size 就推理，兼顾延迟和吞吐；（2）**分级服务**——实时接口用小batch/单条，离线接口用大batch；（3）**多模型实例**——高优先级请求走单条实例，低优先级走批量实例。Triton Inference Server 原生支持动态批处理。
